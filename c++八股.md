# 基础

#static作用

​	static修饰 函数内变量 改变存储位置，栈区->静态存储区，改变生命周期，函数调用结束->程序退出，初始化，只在第一次调用的时候初始化，后续不再初始化，作用域不变，比如统计一个函数被调用了多少次

​	static修饰 全局变量/函数  存储位置、生命周期都不变，仅在本编译单元可见，外部文件无法访问，避免多文件同名函数的命名污染

​	static修饰类成员 变量/函数 属于整个类而非某个对象，类的所有对象共享一份实例，无this指针，必须在类内定义，类外初始化，不能访问类的非静态成员变量/方法

#容器，迭代器，算法，之间的关系

| 组件       | 作用                                                         | 核心头文件                                    |
| ---------- | ------------------------------------------------------------ | --------------------------------------------- |
| **容器**   | 存储数据的通用数据结构（如数组、链表、哈希表）               | `<vector>`, `<list>`, `<map>` 等              |
| **迭代器** | 连接容器与算法的 “桥梁”，提供统一的访问容器元素的接口，算法不需要知道容器的底层结构 | 容器头文件中内置（如`vector<int>::iterator`） |
| **算法**   | 通用的操作数据的函数（如排序、查找、遍历），与容器类型无关   | `<algorithm>`, `<numeric>` 等                 |

#全局变量的目的是？

全局变量的核心目的是**跨作用域 / 跨模块共享数据**，简化高频数据的访问和传递；使用时需重点关注线程安全、命名冲突和代码耦合问题

#指针和引用踩坑有哪些？

访问悬空指针触发段错误、程序崩溃或未定义行为、 空指针解引用、指针声明后未赋值，指向内存随机地址，访问时可能修改系统内存，导致程序崩溃 / 数据损坏，同一堆内存被多次`delete`，触发内存双重释放错误，破坏堆结构，忘记释放指针，内存泄漏；

引用声明时必须绑定对象，未初始化的引用会直接编译失败（指针可未初始化，引用不行);引用绑定到临时对象（如函数返回值、表达式结果），临时对象销毁后引用变为悬空引用;引用一旦绑定对象，无法更改绑定关系，

| 特性     | 指针                     | 引用                     |
| -------- | ------------------------ | ------------------------ |
| 初始化   | 可未初始化（野指针风险） | 必须初始化               |
| 绑定关系 | 可多次指向不同对象       | 绑定后不可更改           |
| 空值     | 可指向 nullptr           | 无空引用（必须绑定对象） |
| 算术运算 | 支持（指针算术）         | 不支持                   |
| 解引用   | 需要 * 符号              | 直接使用（隐式解引用）   |

#成员函数，const在前面在后面有什么区别

const 修饰在前面表示 该函数的返回值不能被修改，对于指针和引用，禁止通过这个返回值修改原对象的成员，const修饰在后边表示 不能通过改函数来修改类内的非静态成员变量     (mutable修饰的变量,缓存、计数的变量，可以被修改,静态成员变量也可以被修改)，本质 const T* const this 左定值，右定向

#vector性能优化，什么时候性能较高

`vector` 的底层是**连续内存**，优势是随机访问（`[]`/`at()`）效率极高（O (1)），但短板是：

1. 内存不足时会触发**重新分配**：申请新内存 → 拷贝 / 移动旧元素 → 释放旧内存（O (n) 开销）；
2. 频繁插入 / 删除（尤其是头部 / 中间）会导致大量元素移动（O (n) 开销）；
3. 不必要的元素拷贝（比如按值传递、未用移动语义）会额外消耗资源。

`vector` 的连续内存布局完美适配 CPU 缓存（缓存行预取），因此**随机访问、顺序遍历**时性能远高于非连续容器（如 `list`/`unordered_map`）。

`vector` 的 `push_back`/`emplace_back` ，只要不触发内存重分配，是O (1) 复杂度。

​	`	emplace_back` 直接在容器内存中构造元素，而 `push_back` 先构造元素，再移动 / 拷贝到容器 ，底层原理是可变参数模板+完美转发，接收任意数量 / 类型的参数；将参数原封不动地传递给元素的构造函数；

`vector` 默认扩容策略是**倍数增长**（通常是 2 倍），每次扩容都会触发内存重分配 + 元素拷贝。提前用 `reserve(n)` 分配足够空间，就能彻底避免扩容开销。

`vector` 的元素若支持移动构造 / 移动赋值，在扩容、插入时会优先使用移动而非拷贝，大幅降低开销（尤其是大对象，如 `string`、自定义类）。

#内存泄漏如何定位？

top指令看到程序运行中内存占用持续上涨且不回落,长时间运行出现 bad_alloc 异常（内存耗尽）,循环 / 网络请求等频繁申请释放内存场景内存增长明显

​	new/delete 不匹配：new [] 必须用 delete [] 释放，new 必须用 delete 释放；

​	条件分支遗漏释放：if/else、try/catch 中部分分支忘记释放内存

​	全局 / 静态指针未释放：全局指针指向的堆内存程序退出前未释放；

​	类析构函数未处理资源：类内申请的内存析构函数未释放；

​	智能指针循环引用：std::shared_ptr 相互引用导致引用计数无法归零

项目不大的时候，手动可以靠日志埋点，检查申请的地方内存是否释放，建议使用智能指针代替裸指针

​	内存数据被篡改，出现随机值 / 逻辑错误（无崩溃但结果异常）

​	堆内存越界可能触发 malloc/free 报错，栈越界可能导致函数返回地址被篡改

​	循环边界：for 循环终止条件是否错误（如 i<=n 而非 i<n）

​	空指针 / 野指针：解引用前未检查指针是否有效（如 delete p 后仍解引用 p）

动态多态：运行期确定 虚函数和继承 

​	虚函数，在基类的前加上virtual关键字，在派生类重写该函数，运行时会根据对象的类型来调用相应的函数，原理：早绑定：编译器编译时已经确定对象调用函数的地址，晚绑定：i.若类使用virtual函数，则编译器会为该类生成虚函数表(一维数组)，存放了虚函数的地址，类对象构造时候会初始化指向该虚表的指针，存放在类对象的最前边，ii.虚函数表指针是在构造函数的时候初始化的

​	多继承的情况下 ，派生类会为每一个包含虚函数的直接基类都维护一个独立的虚函数表指针 每个虚函数表指针都指向对应类的虚函数表，派生类重写了父类的虚函数，就会在虚函数表中将父类的虚函数地址替换成派生类的虚函数/函数地址

​	菱形继承  A-》B     A-》C    B、C-》D     i.虚继承的方式解决菱形继承二义性的问题，编译器会为每一个虚继承的类对象生成虚基类表，存储当前对象到共享基类的偏移量 ii.此时D中存储顶层的基类成员只有一份，虚函数指针数量 等于 包含独立虚函数表的基类数量

#虚函数表、虚表指针的创建时机

​	虚函数表是在编译的时候 发现类中含有virtual修饰的函数，生成虚函数表(一维数组)，存放了虚函数的地址

​	虚表指针是在类对象构造的时候 初始化虚表指针 将虚表指针指向虚函数表的地址，继承的情况下会先调基类的构造，虚表指针指向基类的虚函数表地址，接着调子类的构造，虚表指针指向 子类的虚函数表地址

#构造函数可以是虚函数吗，析构函数呢？

​	构造函数不可以是虚函数，虚函数的实现依赖「虚函数表（vtable）」，而对象的虚表指针是在构造函数执行阶段初始化的,如果构造函数是虚函数，调用时需要通过虚表指针找对应的虚函数，但此时对象还没构造完成，虚表指针都还没初始化，根本无法完成虚函数调用；从语义上，构造函数是「创建特定类型的对象」，调用者必须明确知道要构造的是哪个类的对象，不存在「运行期动态决定构造哪个类」的需求。

​	析构函数建议声明成虚函数，当通过基类指针 / 引用释放派生类对象时，如果析构函数不是虚函数，会触发「早绑定」—— 编译器只根据指针的声明类型（基类）调用基类析构函数，导致派生类的析构函数不执行，进而造成资源泄漏。声明成虚函数，运行期根据对象实际类型（`Derived`），先执行派生类析构释放资源，再执行基类析构。如果基类是「抽象类」（不能实例化），可以声明纯虚析构，(注意：纯虚析构依然需要提供实现（因为派生类析构会调用基类析构))：

#纯虚函数会出现在虚函数表中吗

​	纯虚函数**会出现在虚表中**，表现为 “占位符（陷阱函数）”，子类重写后替换为实际函数地址；若子类未重写纯虚函数，调用时会触发崩溃（提示 “调用纯虚函数”），避免未定义行为。

# 系统编程

#内核态和用户态

​	应用程序无法直接操作硬件（如读写文件、分配内存、网络通信），必须通过**系统调用（System Call）** 向内核 “求助”，这时候就需要从用户态切换到内核态，完成操作后再切回用户态。切换过程需要：保存 / 恢复寄存器（上下文切换）；内存地址空间切换（MMU 重新映射）；权限检查、指令跳转。

这些操作会消耗 CPU 时间（通常几十到几百纳秒），如果程序频繁触发系统调用（如循环调用`read()`读取小数据），切换开销会显著影响性能。

#零拷贝技术

​	Linux 零拷贝（Zero-copy）的核心是消除用户态与内核态之间的冗余数据拷贝、减少 CPU 参与，让数据尽量在内核空间与硬件间流转，大幅提升 I/O 吞吐量并降低 CPU 开销。
​	传统 I/O 的性能瓶颈（以文件发网络为例）DMA(允许外设（如硬盘、网卡、显卡）直接读写系统内存，而不需CPU干预的硬件技术)
传统 read() + write() 流程：

    DMA 从磁盘 → 内核（PageCache）
    CPU 拷贝：内核 → 用户缓冲区（上下文切换）
    CPU 拷贝：用户 → Socket 缓冲区（上下文切换）
    DMA 从 Socket 缓冲区 → 网卡
    总开销：4 次拷贝（2 次 CPU）+ 4 次上下文切换，CPU 被大量占用在数据搬运。

​	1.mmap + write（内存映射）

​	原理：用 mmap() 将文件映射到用户虚拟地址，内核与用户共享同一块物理内存，省去 “内核→用户” 的 CPU 拷贝。
流程： DMA：磁盘 → 内核       mmap 映射：内核页缓存 ↔ 用户空间（共享）   write ()：CPU 拷贝内核页缓存 → Socket 缓冲 区    DMA：Socket 缓冲区 → 网卡
开销：3 次拷贝（1 次 CPU）+ 4 次上下文切换,适用：需要在用户态修改数据的场景（如文件编辑后发送

​	2.sendfile

​	原理：专用系统调用，直接在内核空间完成 “文件描述符 → Socket 描述符” 的数据搬运，完全绕过用户空间。
流程：  DMA：磁盘 → 内核页缓存          CPU 拷贝：内核页缓存 → Socket 缓冲区                 DMA：Socket 缓冲区 → 网卡
开销：3 次拷贝（1 次 CPU）+ 2 次上下文切换   适用：纯文件转发（如 Nginx、Kafka 日志传输），不支持用户态修改数据

​	3.sendfile + DMA Scatter/Gather（真正零 CPU 拷贝，Linux 2.4+）

原理：网卡支持 SG-DMA 时，内核不拷贝数据，只传递内存地址 / 长度描述符；网卡 DMA 直接从内核页缓存读取并发送。
流程：   DMA：磁盘 → 内核页缓存        内核传递描述符到 Socket 缓冲区（极小开销） SG-DMA：网卡直接从内核页缓存读取 → 发送
开销：2 次 DMA 拷贝（0 次 CPU 拷贝）+ 2 次上下文切换

​	4.splice

​	原理：基于内核管道（Pipe），在两个文件描述符间高效移动数据，全程内核态、不经过用户空间。
​	特点：比 sendfile 更通用，可用于文件 ↔ 文件、文件 ↔ Socket、Socket ↔ Socket
​	开销：0 次 CPU 拷贝，仅 DMA 与内核管道操作

#进程之间通信

管道（匿名管道）命名管道（FIFO）消息队列  共享内存 信号量 信号 套接字（Socket）

管道（匿名管道）单向通信，只能在有血缘关系的进程间使用，比如父子进程。基于文件描述符，半双工，数据只能从写端流向读端。底层是内核缓冲区，大小固定，读写会阻塞。只能父子 / 兄弟进程用、单向、面向字节流、阻塞特性。

命名管道（FIFO）解决匿名管道无血缘关系不能通信的问题。在文件系统中有对应的管道文件，不占磁盘空间，只是接口。任意进程可通过打开这个文件实现通信。特点：半双工、面向字节流、可无亲缘关系进程通信。

消息队列是内核维护的链表，按消息类型存取。进程可以按类型发送和接收消息，支持指定类型读取。生命周期随内核，不随进程退出消失，需要手动删除。

优点：异步通信，支持消息边界，不粘包。缺点：容量有限，数据拷贝两次，不适合大数据。

共享内存多个进程直接共享同一块物理内存，是**最快**的 IPC 方式。因为不需要内核拷贝数据，直接读写内存。需要配合同步机制使用，比如信号量、互斥锁，否则会出现竞争。生命周期随内核，效率最高。

信号量本质是计数器，用于实现进程间同步与互斥。P 操作申请资源，V 操作释放资源。二元信号量可以当互斥锁使用，防止多个进程同时访问临界区。主要作用是同步，不是用来传输大量数据。

信号，软中断，是异步通信方式，简单但携带信息少。只能传递信号类型，不能传递复杂数据。用于通知进程发生了某种事件，比如 kill、Ctrl+C。适用简单通知，不适合大数据传输。

套接字（Socket）支持跨主机、跨网络的进程通信。既可本地通信，也可网络通信，是最通用的 IPC。分为 TCP 和 UDP，面向连接 / 无连接，可靠 / 不可靠。服务器、客户端通信几乎都用 Socket。

# 高级特性

#为什么引入智能指针
    C++内存管理中 使用new分配内存，必须在合适的时机释放，忘记释放会造成内存泄漏，释放后没有置为nullptr变成野指针,多个指针指向同一内存，多次释放程序崩溃，智能指针利用RALL机制，资源获取即初始化，将对内存的管理和智能指针对象的生命周期绑定在一起，智能指针对象创建的时候分配内存，智能指针对象销毁的时候，自动释放内存，无需手动delete

#智能指针有哪些

​    auto_ptr c++98引入 已经被弃用 所有权指针 指针对象过期的时候 自动释放指向的内存,缺陷在赋值和拷贝的时候会发生所有权转移 原来的指针变成了空指针,此时访问原来指针是未定义行为 不符合STL容器对元素拷贝语义的要求,auto_ptr不支持数组 它的析构函数调用的是delete 而不是delete[] ,不能使用auto_ptr管理数组
​    unique_ptr 独占所有权指针 同一时间内只允许有一个智能指针指向对象,禁止拷贝和赋值 支持移动语义,支持管理数组 可以使用delete[]释放内存,可以作为函数的返回值高效的把资源返回给被调用者。

​	shared_ptr 共享拥有智能指针，允许多个智能指针指向同一对象 通过引用计数管理对象的生命周期 多一个指向 ++引用计数 当智能指针离开作用域重新赋值的时候 --引用计数 检查引用计数为0自动释放对象,    在堆上分配一个控制块（control block）来管理所指向对象的引用计数和其他相关信息 。当创建一个std::shared_ptr对象并指向一个新的对象时，会同时在堆上分配一个控制块，控制块中包含指向对象的指针、引用计数（初始值为 1）以及可能的自定义删除器等信息

#shared_ptr是线程安全的吗

​	shared_ptr对其管理资源的引用计数是线程安全的,多个线程创建了指向同一资源的shared_ptr的不同智能指针对象进行对引用计数的修改(拷贝 构造 析构 reset等)无需再加锁 

 shared_ptr智能指针对象本身（内部的资源指针和控制块指针）和指向的对象，在多线程访问的情况下，并不是线程安全的

 #什么是循环引用问题
 		带锁的箱子和钥匙的问题，交叉引用之后，不管先析构哪个对象，因为引用计数没办法清0导致不能释放资源
 weak_ptr解决 将一个方向改成weak_ptr 打破循环引用，使用weak_ptr时，需要通过lock函数将其提升为shared_ptr，才能访问对象 。如果对象已经被释放，lock函数会返回一个空的shared_ptr 。

   #make_shared 和shared_ptr的区别     make_shared将对象构造和控制块分配合并为一个原子操作
        1) 内存分配上      make_shared 一次内存分配 同时创建对象和 控制块     shared_ptr 需要两次 内存分配 分别创建对象和控制块 ->内存碎片化
              ）安全性上  make_shared 将对象的构造和智能指针的构造 合并成一个操作 减少显示使用new     避免了对象构造之后，shared_ptr构造之前发生异常，构造对象分配的内存无法正确释放 

#make_shared/shared_ptr 裸指针构造的问题 
    	用同一裸指针多次构造多个 shared_ptr（导致多控制块、引用计数独立）
   	 创建不同的控制块 引用计数不会正常增加 重复析构风险
   	 很多接口（尤其是异步操作、回调注册接口）会要求传入shared_ptr（而非裸指针），
    	目的是通过引用计数管理类对象生命周期，避免对象在函数执行期间被销毁。此时类成员函数内部需要将this转为shared_ptr传入该接口。
    	如果直接传递this指针 ，引用计数不能正确增加 ，
    	如果类对象提前销毁 对象先于异步线程析构  异步线程访问类对象就会访问到悬空指针
   	 而且在析构的时候 还会导致重复释放 第二次析构访问控制块的时候会访问到悬空指针 因为引用计数和资源没有正确关联起来  

​		解决方法：C++11 提供`std::enable_shared_from_this<T>`基类，专门解决 “从`this`指针安全生成`shared_ptr`” 的问题，核心是让对象关联**唯一的控制块**，所有通过`this`生成的`shared_ptr`共享同一个引用计数。原理：通过内部 weak_ptr 绑定对象的唯一控制块，让 this 生成的 shared_ptr 复用该控制块，保证引用计数正确，从而安全管理对象生命周期。

```c++
#include <iostream>
#include <memory>
#include <thread>
#include <chrono>

// 关键：继承std::enable_shared_from_this<类名>
class MyClass : public std::enable_shared_from_this<MyClass> {
public:
    MyClass() {
        std::cout << "MyClass 构造" << std::endl;
    }

    ~MyClass() {
        std::cout << "MyClass 析构" << std::endl;
    }

    // 异步操作接口：需要传入shared_ptr管理生命周期
    void asyncTask() {
        // 错误写法：用this构造shared_ptr，生成新控制块
        // std::shared_ptr<MyClass> ptr(this); 

        // 正确写法：调用shared_from_this()，复用已有控制块
        std::shared_ptr<MyClass> self = shared_from_this();
        
        // 模拟异步线程：持有self，保证对象生命周期
        std::thread t([self]() {
            std::this_thread::sleep_for(std::chrono::seconds(2));
            std::cout << "异步线程执行中，对象地址：" << self.get() << std::endl;
        });
        t.detach(); // 线程后台执行
    }
};

int main() {
    // 1. 先通过shared_ptr创建对象（必须！否则shared_from_this()会崩溃）
    std::shared_ptr<MyClass> obj = std::make_shared<MyClass>();
    
    // 2. 调用成员函数，触发异步操作
    obj->asyncTask();

    // 3. 主线程等待，验证对象生命周期
    std::cout << "主线程等待3秒..." << std::endl;
    std::this_thread::sleep_for(std::chrono::seconds(3));
    
    return 0;
}

```
