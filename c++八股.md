# 内存管理

#内存泄漏如何定位？

top指令看到程序运行中内存占用持续上涨且不回落,长时间运行出现 bad_alloc 异常（内存耗尽）,循环 / 网络请求等频繁申请释放内存场景内存增长明显

​	new/delete 不匹配：new [] 必须用 delete [] 释放，new 必须用 delete 释放；

​	条件分支遗漏释放：if/else、try/catch 中部分分支忘记释放内存

​	全局 / 静态指针未释放：全局指针指向的堆内存程序退出前未释放；

​	类析构函数未处理资源：类内申请的内存析构函数未释放；

​	智能指针循环引用：std::shared_ptr 相互引用导致引用计数无法归零

项目不大的时候，手动可以靠日志埋点，检查申请的地方内存是否释放，建议使用智能指针代替裸指针

#####  使用专业工具**Valgrind（Linux/macOS，免费）**

- 编译：g++ -g -O0 -o test test.cpp
- 运行：valgrind --leak-check=full ./test
- 输出关键信息：definitely lost（确定泄漏）、内存大小、具体行号

#指针越界如何定位？

​	程序崩溃（Segmentation fault/SIGSEGV），核心转储（core dump）

​	内存数据被篡改，出现随机值 / 逻辑错误（无崩溃但结果异常）

​	堆内存越界可能触发 malloc/free 报错，栈越界可能导致函数返回地址被篡改

##### 自查

​	数组访问：检查下标是否超出数组长度（如 int arr [5]; arr [5] 越界）

​	指针算术：指针加减后是否指向分配内存外（如 char* p = new char [10]; p+10 指向越界）

​	循环边界：for 循环终止条件是否错误（如 i<=n 而非 i<n）

​	空指针 / 野指针：解引用前未检查指针是否有效（如 delete p 后仍解引用 p）

#多态如何实现？

静态多态： 编译期确定      函数重载、 模板(类模板和函数模板)，

​	函数重载，指的是允许在同一作用域声明多个功能类似的同名函数，依靠函数参数个数、类型、顺序不同来区别，不能通过返回值类型区别，(隐式类型转化、很多函数没有返回值)，函数重载的原理是函数名修饰，编译过程分为 预处理:头文件中的函数声明拷贝到源文件，避免编译阶段找不到函数定义)，编译:语法分析，同时进行符号汇总(函数名 )，汇编:生成函数名到函数地址的映射，方便之后通过函数名找到函数定义的位置 去执行函数，链接:将多个文件中的符号表汇总合并，查看符号表：objdump -t [二进制文件]    _Z + 类长度 +类名+函数名长度+函数名+E+参数类型首字母

​	类模板和函数模板 ，定义一种通用形式，某些类型是参数化的，编译期通过调用的实际类型参数生成对应的具体代码，运行期执行生成好的代码

动态多态：运行期确定 虚函数和继承 

​	虚函数，在基类的前加上virtual关键字，在派生类重写该函数，运行时会根据对象的类型来调用相应的函数，原理：早绑定：编译器编译时已经确定对象调用函数的地址，晚绑定：i.若类使用virtual函数，则编译器会为该类生成虚函数表(一维数组)，存放了虚函数的地址，类对象构造时候会初始化指向该虚表的指针，存放在类对象的最前边，ii.虚函数表指针是在构造函数的时候初始化的

​	多继承的情况下 ，派生类会为每一个包含虚函数的直接基类都维护一个独立的虚函数表指针 每个虚函数表指针都指向对应类的虚函数表，派生类重写了父类的虚函数，就会在虚函数表中将父类的虚函数地址替换成派生类的虚函数/函数地址

​	菱形继承  A-》B     A-》C    B、C-》D     i.虚继承的方式解决菱形继承二义性的问题，编译器会为每一个虚继承的类对象生成虚基类表，存储当前对象到共享基类的偏移量 ii.此时D中存储顶层的基类成员只有一份，虚函数指针数量 等于 包含独立虚函数表的基类数量

#虚函数表、虚表指针的创建时机

​	虚函数表是在编译的时候 发现类中含有virtual修饰的函数，生成虚函数表(一维数组)，存放了虚函数的地址

​	虚表指针是在类对象构造的时候 初始化虚表指针 将虚表指针指向虚函数表的地址，继承的情况下会先调基类的构造，虚表指针指向基类的虚函数表地址，接着调子类的构造，虚表指针指向 子类的虚函数表地址

#构造函数可以是虚函数吗，析构函数呢？

​	构造函数不可以是虚函数，虚函数的实现依赖「虚函数表（vtable）」，而对象的虚表指针是在构造函数执行阶段初始化的,如果构造函数是虚函数，调用时需要通过虚表指针找对应的虚函数，但此时对象还没构造完成，虚表指针都还没初始化，根本无法完成虚函数调用；从语义上，构造函数是「创建特定类型的对象」，调用者必须明确知道要构造的是哪个类的对象，不存在「运行期动态决定构造哪个类」的需求。

​	析构函数建议声明成虚函数，当通过基类指针 / 引用释放派生类对象时，如果析构函数不是虚函数，会触发「早绑定」—— 编译器只根据指针的声明类型（基类）调用基类析构函数，导致派生类的析构函数不执行，进而造成资源泄漏。声明成虚函数，运行期根据对象实际类型（`Derived`），先执行派生类析构释放资源，再执行基类析构。如果基类是「抽象类」（不能实例化），可以声明纯虚析构，(注意：纯虚析构依然需要提供实现（因为派生类析构会调用基类析构))：
