###### lock-free ringbuffer

```c++
#pragma once
#include <iostream>
#include <array>
#include <atomic>
#include <utility> // 支持std::move
template<class T,std::size_t N>

class SpscRingQueue{
    static_assert(N & (N - 1)==0,"N must be power of two");
public:
    SpscRingQueue(const SpscRingQueue&) = delete;
    SpscRingQueue& operator=(const SpscRingQueue&) = delete;
    SpscRingQueue(SpscRingQueue&& ) = delete;
    SpscRingQueue& operator=(SpscRingQueue&&) = delete;
    // memory order 
    // 编译器和cpu为了优化性能，可能做指令重排/延迟传播到其他核心 ，单线程语义不受影响，多线程环境下，可能出现其他线程观察到的执行顺序和源码顺序不一致
    // 比如SPSC中 生产者先加入数据，再移动指针，消费者可能观察到指针已经更新了，数据没有更新，结果拿到的数据不是最新数据，因此需要使用内存序来对多线程环境下对共享内存的重排/可见性做约束
    // atomic<T>.load/story 里 relaxed 不做任何约束     release-acquire 其他线程可见顺序上，保证release之前的读写在acquire后都可见，且release 不会重排到之前读写操作的前面，acquire不会重排到之后读写操作的后面
    bool Push(T& val) {
        size_t t = tail_.load(std::memory_order_relaxed);
        size_t next = (t + 1) & mask_;
        if(next == head_.load(std::memory_order_acquire)) {
            return false;
        }
        buffer_[t] = val;
        tail_.store(next,std::memory_order_release);
        return true;
    }
    bool Push(T&& val) {
        size_t t = tail_.load(std::memory_order_relaxed);
        size_t next = (t + 1) & mask_;
        if(next == head_.load(std::memory_order_acquire)) {
            return false;
        }
        buffer_[t] = std::move(val);
        tail_.store(next,std::memory_order_release);
        return true;
    }

    bool Pop(T* out) {
        if(out == nullptr) return false;
        size_t h = head_.load(std::memory_order_relaxed);
        if(h == tail_.load(std::memory_order_acquire)) return false;
        *out = buffer_[h];
        head_.store((h + 1)  & mask_,std::memory_order_release);
        return true;
    }

    bool empty() {
        return tail_.load(std::memory_order_acquire) == head_.load(std::memory_order_acquire);
    }

    size_t size() {
        size_t h = head_.load(std::memory_order_acquire);
        size_t t = tail_.load(std::memory_order_acquire);
        return (t - h) & mask_;
    }


private:
    static constexpr size_t mask_ = N - 1;               // static constexpr 编译期常量    相比模运算 位运算更快
    alignas(64) std::array<T,N> buffer_{};               // alignas(64) 防止伪共享 这些变量进入不同的cacheline
    alignas(64) std::atomic<size_t> head_ = 0;            
    alignas(64) std::atomic<size_t> tail_ = 0;
};

```

###### Windows Event

​	在 Windows 下用 Event 让消费者在队列为空时阻塞等待，避免自旋导致 CPU 空转。生产者入队后通过 `SetEvent` 唤醒消费者。为了避免高频数据下每条都触发内核唤醒，我用了一个 `std::atomic<bool> notified` 做边沿触发：只有当 `notified` 是预期的false的时候，（CAS）把它置为到 true 时才 `SetEvent`，表示这是从空闲到活跃的第一次唤醒。消费者被唤醒后会批量 pop 队列；当确认队列已空时，把 `notified` 复位为 false，再进入 `Wait`。这样把唤醒次数降低，消费者线程不再一直循环检查队列，而是把线程挂起睡眠；当生产者放入新数据时，发一个通知把消费者唤醒。Event 是 Windows 内核里的一个同步对象。

- **消费者**调用 `WaitForSingleObject(event, ...)`：如果 event 没触发，它就睡眠，不占 CPU。
- **生产者**调用 `SetEvent(event)`：触发 event，把一个或多个正在等待的线程唤醒。

- **auto-reset（自动复位）**：一次通知只唤醒一次”，唤醒后系统自动把事件清掉。
- CAS 在这里用来做边沿触发：只有把 `notified_` 从 false 原子改为 true 的那个生产者线程才会 SetEvent，其它生产者 CAS 失败就直接返回，从而避免高频数据下重复唤醒带来的内核开销。
- `compare_exchange_strong` : 因为不写自旋重试循环；`weak` 可能出现“伪失败”（值其实相等也返回 false），如果不用循环会导致该唤醒时没唤醒。`strong` 不会有这种伪失败，所以在这种“一次判断决定是否唤醒”的场景更稳、更直观。

###### moodycamel::ConcurrentQueue

​	1.`moodycamel::ConcurrentQueue` 中的 `token`（令牌），为每个线程分配专属的访问凭证，队列会为持有 token 的线程缓存局部状态（比如可用的内存块、队列分段信息）；避免多个线程争抢同一个全局原子变量，将竞争分散到线程本地，大幅提升高并发下的入队 / 出队效率。

​	2.多生产者下顺序能保证吗？多生产者并发入队无法保证全局严格时间顺序（取决于线程调度），只能保证“入队发生的先后关系”。但通常能保证单生产者内部的相对顺序（同一线程 enqueue 的顺序）。数据有时间戳，消费端重对齐

​	3.事件通知 + notified CAS：会不会漏唤醒？`notified` 是边沿触发：空→非空只唤醒一次，避免 2000/s 每条都进内核。consumer 在 确认队列空时复位 `notified=false` 再 Wait，降低漏唤醒概率。auto-reset 适合单消费者一次 SetEvent 唤醒一次，自动复位，避免手动 ResetEvent 出错。

**序列化：针对高频传感器数据采取优化策略，提高吞吐与传输效率，有效缓解高频数据在队列中积压情况；**

​	批处理 + RapidJSON 在高频场景更快的核心原因是它支持流式 Writer 直接写入 buffer，避免构建 DOM 树和大量小对象分配；配合 Reserve/复用 buffer，可以显著减少 heap 分配和递归遍历开销，所以在动捕这种 joints  很多、每一批数据 很大的序列化上优势更明显。

**Kafka发送: 序列化之后的数据，通过librdkafka C++ Producer发出供下游使用,采取合适的配置策略提升吞吐和可靠性**

> 我们把采集线程和发送线程解耦，发送线程批量序列化后用 librdkafka C++ Producer 异步 produce。核心目标是：吞吐足够高、失败可控、不会因为 Kafka 拥塞把上游拖死。实现上我做了三件事：**批量消息**（减少 produce 次数）、**正确驱动 producer 的 poll/flush**（保证 delivery report 和 outq 释放）、以及**吞吐/可靠性参数调优**（linger/batch、acks/retries、压缩、幂等可选），并通过 outq_len、queue_full、delivered_ok/fail 做可观测闭环。

## A. 为什么 producer 必须 `poll()`？

> librdkafka 的 Producer 是异步 IO，`produce()` 只是把消息放进内部队列（outq）。`poll()` 会驱动网络 IO、处理 delivery report 回调、释放 outq。如果 outq 满了你不 poll，`ERR__QUEUE_FULL` 会持续出现，发送线程就会卡住，最终导致本地队列积压。

## B. 为什么要 batch 后再 produce？

- 大量小消息会产生高系统调用/协议开销
- batch 后 produce 次数下降，吞吐明显提升
- 你还能用 linger/batch.size 进一步让 broker 端批处理更有效

# 3) “配置策略”怎么讲（吞吐 vs 可靠性分层）

下面是面试最常考的：你要能说清楚**哪些配置是吞吐向，哪些是可靠性向，以及它们的 trade-off**。

------

## 3.1 吞吐向配置（Throughput knobs）

你可以按这 4 个点讲：

### 1) 批处理与等待：`linger.ms` / `batch.num.messages` / `batch.size`

- **作用**：让 producer 在短时间内攒一批再发，减少请求次数，提高压缩比和吞吐
- **代价**：增加少量端到端延迟（通常毫秒级）
- **口径**：我们本身已经做了应用层 batch；producer 侧再通过 linger 做微批，进一步提升吞吐

### 2) 压缩：`compression.type`（lz4 / zstd）

- **作用**：降低网络带宽和 broker 写入压力，吞吐通常更稳
- **代价**：增加 CPU；要看瓶颈在网络还是 CPU
- **口径**：动捕 payload 大时压缩收益更明显；眼动/EEG 小 payload 压缩收益有限

### 3) 内部队列容量：`queue.buffering.max.messages` / `queue.buffering.max.kbytes`

- **作用**：允许 producer 在短暂网络抖动时缓存更多数据
- **代价**：内存占用上升；不可无限大，否则 OOM 风险
- **口径**：我们配合应用层软上限/丢弃策略，避免 Kafka 慢时内存失控

### 4) 并行与网络：`socket.send.buffer.bytes` 等

- **作用**：在高带宽/高 RTT 场景改善吞吐
- **口径**：不是第一优先，一般压测发现网络瓶颈再调

------

## 3.2 可靠性向配置（Reliability knobs）

> acks参数控制生产者发送消息的确认机制，也就是控制生产者发送消息，Broker端确认成功写入的条件
>
> 消息写入kafka 过程:
>
> 对于某个topic-partition 
>
> 1.生产者将消息发给该分区的leader broker
>
> 2.leader broker将消息追加本地日志 更新内存/页缓存（不等磁盘fsync
>
> 3.leader把数据复制给follower
>
> 4.follower 复制完成后返回leader
>
> 5.leader根据acks规则决定什么时候返回生产者 suc/fal
>
> acks = 0
>
> 生产者不等任何broker响应，发出去就当成功
>
> acks = 1
>
> leader写入本地日志后就返回成功，不等follower复制
>
> （**已 ack 仍可能丢**：如果 leader 写入后立刻宕机，而 follower 还没复制到，这条消息可能不在新 leader 上 → 丢失）
>
> acks = all
>
> leader需要等待ISR集合里所有的副本都确认写入才返回成功
>
> （ISR是与leader同步的一组副本集合，如果落后太多会被踢出）
>
> （min.insync.replicas 指的是ISR中至少的副本数量，ack =all 时候，min.insync.replicas > 1,这样至少两个副本写成功才ack，leader宕机不丢）
>
> 消息重复问题
>
> 网络抖动导致生产者没有收到ack 生产者认为失败并重试，其实broker写入成功了，生产者重试写入，导致消息重复
>
> 解决
>
> enable.idempotence=true  acks=all 幂等producer会给每个分区的消息带序列号，broker可以识别并去重
>
> ##### 项目里如何ack？
>
> 使用librdkafka 异步 producer，`produce()` 只代表进入客户端队列，为了确认 broker 端是否按 acks 语义确认写入，我注册了 Delivery Report 回调并在发送线程里持续 `poll()` 驱动回调执行。运行过程中我统计 `produced_ok / delivered_ok / delivered_fail`，并监控 `outq_len` 和 `QUEUE_FULL` 次数衡量背压。退出时 `flush()` 并输出最终投递成功率和失败原因日志，因此可以用数据证明消息真实投递情况，而不是只停留在“入队成功”。
>
> `acks=all` 只能保证 broker 端在 ISR 副本确认后才返回成功，但仍可能因为 producer 崩溃导致消息没发出去，或者因为 min.insync.replicas 配置过低导致只剩 leader 也会 ack。我们会用 delivery report 确认最终投递结果，并配合 replication.factor>=3、min.insync.replicas>=2 来避免“假成功”。对关键数据进一步用本地 WAL 做兜底，失败时可阻塞重试或降级落盘。

# 4) Backpressure（你必须讲清楚，否则会被追着问）

你现在已经有正确的代码逻辑，面试就这样讲：

> Kafka 的背压通过两层处理：
>
> 1. producer outq 满返回 `ERR__QUEUE_FULL`，我们 `poll` 释放队列并退避，避免忙等；
> 2. 如果 queue_full 持续上升，说明下游不可用或吞吐不足，这时对低优先级高频数据启用降采样/丢弃并打点告警，防止本地队列和内存失控。

再补一句关键细节（你代码也该这么改）：

- 发送重试循环必须检查 `running_`，否则 stop/join 会卡死（你眼动那段 while(true) 就是可改进点）

# 6) 最常见追问与标准答法

### Q：`RK_MSG_COPY` 为什么？能不能零拷贝？

- COPY：最安全，payload 生命周期简单
- 零拷贝：要管理内存直到 delivery report 才释放（buffer pool + RK_MSG_FREE），复杂度更高

### Q：你为什么 `produce` 后还要 `poll(0)`？

- 处理 delivery report/回调，推进状态机、释放资源；否则 outq 释放不及时

### Q：Kafka 长时间不可用会怎样？

- 进入 backpressure：queue_full↑，outq_len↑
- 策略：退避重试 + 熔断/降采样 + 告警；并保证 stop 可终止

**音视频推流：基于 Boost.Beast 实现轻量级 WebSocket Server 进行推流；方便后端实时观看视频和音频数据**

> 我们用 Boost.Beast 起本地 WebSocket Server 做实时预览旁路。视频不走复杂的媒体协议栈，而是把每帧编码成 JPEG 后通过 WS binary frame 推送；音频推 PCM 小块数据。这样后端/前端实现简单、端到端延迟低，适合调试与实时观察。推流链路和 Kafka 主链路解耦，网络慢时采用有界队列和丢帧策略，保证不会反压采集线程。

# 2) 面试官最常问的问题 & 你怎么答

## Q1：为什么视频用 JPEG、音频用 PCM？为什么不用 H264/AAC？

**答法（工程取舍）**

- 目标是“实时预览/监控”，不是生产级直播；优先 **易对接、低耦合、快速落地**。
- **JPEG**：浏览器/后端解码简单，前端直接 `<img>`/canvas 画；不需要解码器和复杂封装。
- **PCM**：直接播放/分析简单，编码延迟最小；AAC/Opus 需要额外编码器和封包、还要处理缓冲和同步。
- 代价我们也清楚：带宽更大、压缩效率差，所以只用于内网或低并发预览；若要公网/高并发，会升级 H264/Opus/WebRTC。

> 这句很关键：你承认 trade-off，并给出演进方向。

------

## Q2：WebSocket 上传的消息格式是什么？怎么区分 JPEG/PCM？

你必须讲出一个**轻量协议**，哪怕是你自己定义的：

推荐口径（非常容易实现也容易讲）：

- WebSocket **binary frame**
- 自定义固定头 + payload
  - `type`：1=video_jpeg, 2=audio_pcm, 3=meta
  - `codec`：JPEG / PCM（可选）
  - `ts_us`：时间戳（微秒）
  - `seq`：序号
  - `payload_len`
- payload：
  - JPEG：完整 jpeg bytes（包含 SOI/EOI）
  - PCM：一段 PCM samples（例如 10ms/20ms 的 chunk）

面试一句话：

> 我们用 type+ts+seq 的帧头把音视频统一封包，方便解码、同步和丢包检测。

------

## Q3：延迟怎么控制？网络慢怎么办？

这块是你真正的“工程能力”：

- **有界队列**：每个 session 有独立发送队列，设上限（例如最多缓存 200ms~500ms 的帧）
- **丢帧策略（视频优先丢）**：
  - 队列满：丢旧 JPEG，只保留最新帧（保证实时画面跟得上）
  - 音频可以更严格：只缓存很短的 PCM（例如 100ms），过期丢弃或静音填补
- **发送异步化**：采集线程只 enqueue，发送在线程/IO loop，避免网络阻塞拖慢采集

> 面试官会问“为什么丢旧不丢新”：实时预览看最新状态，宁可跳帧不延迟。

------

## Q4：多客户端会不会互相影响？

推荐答法：

> 每个 WebSocket session 独立队列和背压。慢客户端只会丢自己的帧，不影响其他客户端，也不反压采集线程。

------

## Q5：Beast 里怎么写？同步还是异步？为什么要串行写？

关键点必须提：

- `async_write` 必须串行（同一 session 不能并发写）
- 用 `strand` 或 “发送队列 + 只在前一个 write 完成后触发下一个” 保证串行

一句话：

> Beast 的 write 需要串行化，我用 per-session send queue + strand 保证线程安全和顺序。

------

## Q6：音频 PCM 怎么播放？采样率/位深/通道怎么传？

你要能说清“PCM 参数是协议的一部分”：

- sample_rate（比如 16k/48k）
- channels（1/2）
- bit_depth（16-bit 常见）
- endian（一般 little-endian）
- frame duration（比如每包 10ms/20ms）

> 没这些，后端没法正确播放。即便你现在是固定值，也要说“固定配置写死/或通过 meta 帧下发”。

------

## Q7：音视频同步怎么做？

你的方案（JPEG+PCM）同步通常是“软同步”，不用做到严格 lip-sync：

- 两路都带 `ts_us`
- 后端按 ts 做简单对齐播放
- 或以音频为时钟，视频尽量贴近当前音频时间戳

面试口径：

> 预览链路做的是近实时展示，按 ts_us 做软同步即可；真正离线分析在 Kafka/落盘链路做严格对齐。

# 3) 面试“加分点”：你主动提的演进路线

# 4) 你现在可以直接背的一段“2 分钟完整版”

> 我们在采集端用 Boost.Beast 实现 WebSocket Server，提供实时预览旁路，和 Kafka 主链路解耦。视频帧用 JPEG 编码后通过 WS binary frame 推送，音频推 PCM 小块数据。这样前后端接入成本低：视频直接解码 JPEG 显示，音频按采样率/位深播放。为了控制延迟与稳定性，每个连接维护独立的有界发送队列，网络慢时对视频采取丢旧保新策略，音频只保留很短缓冲，避免无界积压影响采集线程。Beast 里采用异步写并串行化写操作（send queue/strand），保证线程安全和顺序。每个包带 type、ts_us、seq，既能区分音视频，也能做简单同步和丢包检测。这个方案适合内网实时观察；如果要上公网或节省带宽，再演进到 H264/Opus 或 WebRTC。
