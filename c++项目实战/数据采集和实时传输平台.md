# 数据采集和实时传输平台

###### 相关技术：Kafka、WebSocket、OpenCV 、FFmpeg、RapidJSON、并发编程、工厂模式、单例模式

###### 项目描述:

​	面向在座舱环境中驾驶员穿戴多种传感器的场景，构建数据接入与实时传输中间层系统，承担多设备数据采集、时间对齐、缓冲、序列化与转发职责，具体的传感器设备包括眼动仪、脑电、动作捕捉、摄像头与音频设备。重点解决多源异构设备接入、并发采集、批量序列化、高吞吐可靠传输与系统稳定性的问题。

###### 主要工作:

**对设备抽象与接入: 抽象SensorDevice 基类，统一initialize/start/stop/getData接口，结合工厂模式实现多设备注册与动态管理，解耦设备SDK差异；**



这部分任务相关的 WebSocket 是做什么的？
做采集端和后端之间的同步控制通道。它承担的是“任务/设备参数查询、采集状态上报、设备信息查询、采集任务设备变更”这类控制面能力。也就是说，音视频推流链路解决的是“实时看见数据”，而这个同步 WebSocket 客户端解决的是“采集任务怎么下发、采集状态怎么回传、设备配置怎么协同”。两个链路都基于 WebSocket，但职责完全不同，一个偏数据面，一个偏控制面。

这个 WebSocketSyncClient 的整体设计思路是什么？
 我的设计思路是把它做成一个轻量的同步请求客户端，每次任务查询、状态上报、设备更新、设备列表查询，发送一条 JSON 请求，读取响应

这个客户端支持哪些业务动作？
 从代码上看，核心支持四类动作。第一类是 `RequestCollectTaskData`，用于采集端向后端查询当前采集任务信息。第二类是 `SendCollectStatus`，用于采集开始和结束时向后端上报采集状态。第三类是 `SendDeviceUpdateRequest`，用于任务执行过程中设备变更上报，比如采集任务绑定设备调整。第四类是 `RequestDeviceData`，用于查询某类设备的可用设备列表。也就是说，这个客户端本质上是一个针对采集业务协议封装好的 WebSocket RPC 客户端。

它的请求-响应流程是怎样的？
 整体流程是固定的。先创建 `io_context`、`resolver` 和 `websocket::stream<tcp::socket>`。然后通过 `resolver.resolve(host_, port_)` 解析目标地址，再 `connect` 到目标主机，之后通过 `ws.handshake(host_, path_)` 完成 WebSocket 握手。握手完成后，把业务请求组织成 JSON，比如 `dataType`、`collectId`、`collectTaskId`、`timestamp` 等字段，然后 `ws.write` 把请求发出去。接下来通过 `ws.read` 读取服务端返回的数据，转成字符串后做 JSON 解析，最后映射成业务结构体，比如 `CollectTaskResponse` 或 `GainDeviceInfo` 列表。处理完响应后主动 `close` 连接。整个流程非常标准，就是一个同步式 WebSocket 请求-响应模型。

为什么这里用 WebSocket 做任务协同，而不是普通 HTTP？
 这个问题本质上是协议选型问题。我这里之所以沿用 WebSocket，是因为整个采集端和后端的协同协议本来就是围绕 WebSocket 建的，任务获取、状态变更、设备信息都走统一的通道，协议格式一致，后端也更容易统一处理。而且 WebSocket 本身也是建立在 TCP 之上的全双工协议，后续如果某些控制消息需要服务端主动推送，也容易扩展。虽然当前实现是同步请求-响应，但协议层选 WebSocket 给了后续扩展空间。相对 HTTP 来说，这种做法协议统一、实现一致，减少了客户端维护两套通信栈的成本。

WebSocketStructs 这一层的作用是什么？

 “协议模型层”。后端返回的是 JSON，但界面和业务逻辑不应该直接操作 JSON，所以我把它解析成明确的 C++ 结构体，比如 `Device`、`Pilot`、`CollectTask`、`CollectTaskResponse`、`GainDeviceInfo` 等。这样做有三个好处。第一，协议字段和业务模型解耦，界面层只面向结构体。第二，字段是否可空通过 `std::optional` 明确表达，比如 `deviceChildId` 这种不是所有设备都有。第三，后续如果 JSON 结构变了，只需要改解析层，不会把变化扩散到整个界面逻辑。

这部分 WebSocket 的优点是什么？
 这套同步 WebSocket 设计的优点主要有四个。第一，协议统一，任务查询、状态上报、设备更新、设备查询都走同一套 WebSocket JSON 通道。第二，业务模型清晰，通过结构体解析把控制协议从界面逻辑中隔离出来。

如果问“这块和音视频推流有什么关系”，你怎么答？
 它们底层都用了 WebSocket，但职责完全不同。音视频推流那块是高频、低延迟、二进制流式数据传输，属于数据面；这个 WebSocketSyncClient 是低频、同步、JSON 请求响应式的控制面。控制面负责采集任务协同和状态回传，数据面负责实时预览。这样拆开之后，协议职责清晰，性能特征也不一样，不会把控制请求和实时流混到一条链路里。

###### lock-free ringbuffer

```c++
#pragma once
#include <iostream>
#include <array>
#include <atomic>
#include <utility> // 支持std::move
template<class T,std::size_t N>

class SpscRingQueue{
    static_assert(N & (N - 1)==0,"N must be power of two");
public:
    SpscRingQueue(const SpscRingQueue&) = delete;
    SpscRingQueue& operator=(const SpscRingQueue&) = delete;
    SpscRingQueue(SpscRingQueue&& ) = delete;
    SpscRingQueue& operator=(SpscRingQueue&&) = delete;
    // memory order 
    // 编译器和cpu为了优化性能，可能做指令重排/延迟传播到其他核心 ，单线程语义不受影响，多线程环境下，可能出现其他线程观察到的执行顺序和源码顺序不一致
    // 比如SPSC中 生产者先加入数据，再移动指针，消费者可能观察到指针已经更新了，数据没有更新，结果拿到的数据不是最新数据，因此需要使用内存序来对多线程环境下对共享内存的重排/可见性做约束
    // atomic<T>.load/story 里 relaxed 不做任何约束     release-acquire 其他线程可见顺序上，保证release之前的读写在acquire后都可见，且release 不会重排到之前读写操作的前面，acquire不会重排到之后读写操作的后面
    bool Push(T& val) {
        size_t t = tail_.load(std::memory_order_relaxed);
        size_t next = (t + 1) & mask_;
        if(next == head_.load(std::memory_order_acquire)) {
            return false;
        }
        buffer_[t] = val;
        tail_.store(next,std::memory_order_release);
        return true;
    }
    bool Push(T&& val) {
        size_t t = tail_.load(std::memory_order_relaxed);
        size_t next = (t + 1) & mask_;
        if(next == head_.load(std::memory_order_acquire)) {
            return false;
        }
        buffer_[t] = std::move(val);
        tail_.store(next,std::memory_order_release);
        return true;
    }

    bool Pop(T* out) {
        if(out == nullptr) return false;
        size_t h = head_.load(std::memory_order_relaxed);
        if(h == tail_.load(std::memory_order_acquire)) return false;
        *out = buffer_[h];
        head_.store((h + 1)  & mask_,std::memory_order_release);
        return true;
    }

    bool empty() {
        return tail_.load(std::memory_order_acquire) == head_.load(std::memory_order_acquire);
    }

    size_t size() {
        size_t h = head_.load(std::memory_order_acquire);
        size_t t = tail_.load(std::memory_order_acquire);
        return (t - h) & mask_;
    }


private:
    static constexpr size_t mask_ = N - 1;               // static constexpr 编译期常量    相比模运算 位运算更快
    alignas(64) std::array<T,N> buffer_{};               // alignas(64) 防止伪共享 这些变量进入不同的cacheline
    alignas(64) std::atomic<size_t> head_ = 0;            
    alignas(64) std::atomic<size_t> tail_ = 0;
};

```

###### Windows Event

​	在 Windows 下用 Event 让消费者在队列为空时阻塞等待，避免自旋导致 CPU 空转。生产者入队后通过 `SetEvent` 唤醒消费者。为了避免高频数据下每条都触发内核唤醒，我用了一个 `std::atomic<bool> notified` 做边沿触发：只有当 `notified` 是预期的false的时候，（CAS）把它置为到 true 时才 `SetEvent`，表示这是从空闲到活跃的第一次唤醒。消费者被唤醒后会批量 pop 队列；当确认队列已空时，把 `notified` 复位为 false，再进入 `Wait`。这样把唤醒次数降低，消费者线程不再一直循环检查队列，而是把线程挂起睡眠；当生产者放入新数据时，发一个通知把消费者唤醒。Event 是 Windows 内核里的一个同步对象。

- **消费者**调用 `WaitForSingleObject(event, ...)`：如果 event 没触发，它就睡眠，不占 CPU。
- **生产者**调用 `SetEvent(event)`：触发 event，把一个或多个正在等待的线程唤醒。

- **auto-reset（自动复位）**：一次通知只唤醒一次”，唤醒后系统自动把事件清掉。
- CAS 在这里用来做边沿触发：只有把 `notified_` 从 false 原子改为 true 的那个生产者线程才会 SetEvent，其它生产者 CAS 失败就直接返回，从而避免高频数据下重复唤醒带来的内核开销。
- `compare_exchange_strong` : 因为不写自旋重试循环；`weak` 可能出现“伪失败”（值其实相等也返回 false），如果不用循环会导致该唤醒时没唤醒。`strong` 不会有这种伪失败，所以在这种“一次判断决定是否唤醒”的场景更稳、更直观。

###### moodycamel::ConcurrentQueue

​	1.`moodycamel::ConcurrentQueue` 中的 `token`（令牌），为每个线程分配专属的访问凭证，队列会为持有 token 的线程缓存局部状态（比如可用的内存块、队列分段信息）；避免多个线程争抢同一个全局原子变量，将竞争分散到线程本地，大幅提升高并发下的入队 / 出队效率。

​	2.多生产者下顺序能保证吗？多生产者并发入队无法保证全局严格时间顺序（取决于线程调度），只能保证“入队发生的先后关系”。但通常能保证单生产者内部的相对顺序（同一线程 enqueue 的顺序）。数据有时间戳，消费端重对齐

​	3.事件通知 + notified CAS：会不会漏唤醒？`notified` 是边沿触发：空→非空只唤醒一次，避免 2000/s 每条都进内核。consumer 在 确认队列空时复位 `notified=false` 再 Wait，降低漏唤醒概率。auto-reset 适合单消费者一次 SetEvent 唤醒一次，自动复位，避免手动 ResetEvent 出错。

**序列化：针对高频传感器数据采取优化策略，提高吞吐与传输效率，有效缓解高频数据在队列中积压情况；**

​	批处理 + RapidJSON 在高频场景更快的核心原因是它支持流式 Writer 直接写入 buffer，避免构建 DOM 树和大量小对象分配；配合复用 buffer，可以显著减少 heap 分配和递归遍历开销，所以在动捕这种 joints  很多、每一批数据 很大的序列化上优势更明显。

**Kafka发送: 序列化之后的数据，通过librdkafka C++ Producer发出供下游使用,采取合适的配置策略提升吞吐和可靠性**

> 我们把采集线程和发送线程解耦，发送线程批量序列化后用 librdkafka C++ Producer 异步 produce。核心目标是：吞吐足够高、失败可控、不会因为 Kafka 拥塞把上游拖死。实现上我做了三件事：**批量消息**（减少 produce 次数）、**正确驱动 producer 的 poll/flush**（保证 delivery report 和 outq 释放）、以及**吞吐/可靠性参数调优**（linger/batch、acks/retries、压缩、幂等可选），并通过 outq_len、queue_full、delivered_ok/fail 做可观测闭环。

问请简述 Kafka 的核心原理（架构、数据流转过程）。
 Kafka 本质上是分布式追加日志系统。架构上有 Broker、Topic、Partition、Producer、Consumer Group，以及负责元数据和选举协调的控制面；在新版本里控制面由 KRaft quorum 负责。数据流转过程可以概括为：Producer 根据 topic、key 和分区策略选择一个 partition，把消息写到该 partition 的 leader；leader 先顺序追加本地日志，再由 follower 从 leader 拉取复制；当满足 ack 条件后返回 producer；consumer 端按 offset 采用拉模式读取消息，处理完成后提交 offset 到 Kafka 内部主题。Kafka 的核心不是“把消息存一下”，而是“把消息组织成可追加、可复制、可回放、可并行消费的分区日志”。

问讲一下你对消息队列的理解（核心价值、适用场景）。
 我对 MQ 的理解是，它本质上是在系统之间引入一个“可缓冲、可异步、可解耦、可回放”的中间层。核心价值不只是异步通信和削峰填谷，还包括上下游解耦、失败隔离、流量缓冲、广播分发、消费位点管理、重试补偿，以及为实时链路提供统一事件总线。适用场景主要是高并发异步化、多系统订阅同一份数据、日志埋点汇聚、实时数仓、任务解耦、以及对回放和审计有要求的系统。Kafka 更适合高吞吐事件流和日志流，RabbitMQ 更适合复杂路由和业务队列，RocketMQ 更偏业务特性丰富的消息平台。

问Kafka 的核心特性有哪些？
 Kafka 的核心特性我会概括成八个词：高吞吐、可持久化、分区并行、多副本高可用、消费组、可回放、顺序写、以及流式处理生态。高吞吐来自顺序写磁盘、批量收发、分区并行和 Page Cache 友好；高可用来自 leader/follower 多副本和 ISR 机制；消费组让同一份数据可以被多个系统独立订阅；可回放让数据既能做在线消费，也能做补数和重放。官方设计文档本身就是按“分布式日志”思路描述 Kafka 的。

问Kafka 是推（Push）模式还是拉（Pull）模式？支持批量处理还是单条处理？
 Kafka 消费端是拉模式，不是 broker 主动推给消费者。producer 和 consumer 两端都天然支持批量处理，不是只能单条。consumer 自己控制拉取节奏、批次大小和 offset，因此能更好地做流控和背压。producer 侧也有 batch 和 linger 相关配置，consumer 侧也有 fetch 批量读取。Kafka 设计文档一直强调它是由消费者主动拉取数据的模型。

问什么是零拷贝？Kafka 的零拷贝是怎么实现的（底层调用的函数 / 机制）？
 零拷贝的核心目标是减少数据在用户态和内核态之间来回复制的次数，降低 CPU 消耗和上下文切换。Kafka 的典型场景是 broker 把磁盘中的日志数据发给 socket，底层会利用操作系统提供的零拷贝机制，例如 Linux 常见的 sendfile 路径，让数据尽量直接从 page cache 进入网卡发送路径，而不是先拷到用户态再拷回内核。Kafka 官方设计文档明确提到它利用零拷贝优化，让数据被拷入 page cache 一次后可以被反复复用给消费者，从而逼近网络带宽上限。

问有了解过 Kafka 分区（Partition）的概念吗？其核心作用是什么？
 Partition 是 Kafka 的核心抽象。一个 topic 在物理上会被拆成多个 partition，每个 partition 本质上是一条有序追加日志。它的核心作用有三个：第一，做水平扩展，把一个 topic 的写入和读取分摊到多个分区；第二，提供消费并行度，一个 partition 在同一个消费组里只能被一个 consumer 实例消费，所以分区数决定并行上限；第三，提供顺序性边界，Kafka 只保证分区内顺序，不保证 topic 全局顺序。

问正常情况下，一个 Topic 下是一个分区还是多个分区？多个分区的好处是什么？
 正常生产环境通常是多个分区，不会只用一个分区。多个分区的好处主要是提升吞吐、提升消费并行度、支持多副本和故障切换、以及缓解热点。代价也很明确：分区一多，元数据管理、rebalance 开销、文件句柄和运维复杂度都会上升，而且顺序只能保证到分区级别，不能保证全局顺序。所以分区数不是越多越好，而是要和吞吐目标、消费实例数、单分区负载一起设计。

问Kafka 的 Leader 和 Follower 分别是干什么用的？
 每个 partition 在任意时刻有一个 leader 和若干 follower。leader 对外提供读写服务，producer 写的是 leader，consumer 默认读的也是 leader；follower 的职责是从 leader 拉取数据并复制，保证副本冗余和故障恢复。当 leader 宕机或不可用时，系统会从合适的副本中选出新的 leader，保证服务可继续。

问请介绍 Kafka 的 ISR 机制（In-Sync Replicas，同步副本集）。
 ISR 是“和 leader 保持足够同步的副本集合”。不是所有副本都永远算同步副本，只有那些复制延迟、同步状态满足要求的 follower 才会进入 ISR。它的意义是把“能否参与高可靠写入确认”和“能否参与 leader 选举”限制在一个更安全的副本集合里。比如 producer 端配置 `acks=all` 时，leader 需要等 ISR 中副本确认后才返回成功，所以 ISR 直接影响可靠性与延迟。Kafka 官方 producer 配置文档里也把 `acks=all` 描述成最强的持久化保证。

问Kafka 用什么维护消费偏移量（Offset）？
 现代 Kafka 主要用内部主题 `__consumer_offsets` 来维护消费组 offset，不再依赖 ZooKeeper。consumer 处理完成后提交 offset，本质上就是向这个内部主题写一条记录。这样做的好处是 offset 管理本身也被纳入 Kafka 的复制、持久化和高可用体系中。

问最新版本还依赖 ZooKeeper 吗？原因是什么？
 如果你面的是较新版本口径，这题一定要答新版本：Kafka 4.0 及更高版本 broker 端已经不再支持 ZooKeeper，只支持 KRaft；官方升级文档明确要求，ZooKeeper 模式集群必须先迁移到 KRaft 才能升级到 4.0+。原因是 Kafka 现在把元数据管理内聚到自己的 KRaft quorum 里，架构更简单，部署和运维成本更低，也减少了双系统协调的复杂度。

问除了异步通信和削峰填谷，使用 MQ 还有哪些考量？
 除了异步和削峰，我还会从六个角度看 MQ。第一是系统解耦，让上下游生命周期独立；第二是失败隔离，避免慢系统直接拖垮主链路；第三是广播分发，同一份消息给多个系统独立消费；第四是重试和补偿，把瞬时失败从同步调用转成可恢复过程；第五是可回放与审计，尤其 Kafka 天然适合做事件重放；第六是流式数据平台能力，比如实时计算、监控埋点、数仓入湖等。

问什么时候会用到 Kafka？其核心应用场景是什么？
 Kafka 的核心应用场景是高吞吐事件流。最典型的是日志收集、埋点、监控指标、IoT 数据、行为数据、实时 ETL、CDC、实时数仓和流处理。它特别适合“同一份数据要被多个系统消费，而且需要保留和回放”的场景。如果只是简单任务队列、复杂路由、小吞吐低延迟业务消息，未必优先选 Kafka。

问Kafka 如何保证高可用性？（结合集群、ISR、Leader 选举等）
 Kafka 的高可用来自三层。第一层是集群层，多 broker 部署，partition 多副本存储。第二层是复制层，leader 负责写，follower 持续复制，ISR 控制哪些副本算同步副本。第三层是选举层，leader 宕机后，controller 会在合适副本中选出新 leader，客户端刷新元数据后继续生产和消费。简单说就是“多副本 + ISR + leader 选举 + 元数据协调”。

问Kafka 如何保证消息不丢失？可能丢失消息的场景有哪些？如何解决？
 消息不丢要分 producer、broker、consumer 三段来看。producer 侧可能因为网络异常、客户端崩溃、未正确 ack 导致丢失；broker 侧可能因为 leader 刚写入还没复制就宕机，或者不安全选举导致丢失；consumer 侧可能因为先提交 offset 再处理业务，结果业务失败但 offset 已前移。对应的解决方案是：producer 配 `acks=all`、`retries>0`、`enable.idempotence=true`；broker 侧用多副本、ISR，避免不安全选举；consumer 侧先处理业务再提交 offset，并做好幂等。官方文档也明确说明 `acks=all` 配合 ISR 能提供最强持久化保证，而幂等在没有冲突配置时默认启用。

问Kafka 如何保证消息的顺序性？（整体顺序、分区内顺序）
 Kafka 只保证分区内顺序，不保证整个 topic 全局顺序。要保证业务顺序，核心办法是把同一业务实体的消息固定发到同一个 partition，比如用 orderId、deviceId 作为 key。Producer 侧还要注意重试乱序问题，官方文档指出，如果幂等关闭且 `max.in.flight.requests.per.connection` 大于 1，重试后有乱序风险；如果开启幂等，则顺序可保留。

问Kafka 如何防止脑裂问题？
 Kafka 防脑裂靠的是严格的领导者和元数据管理机制，而不是让多个副本同时对外写。分区同一时刻只有一个 leader 对外提供服务，leader 变更由 controller 统一协调；可参与选举的副本受 ISR 等规则限制，不是任意 follower 都能当 leader。这样可以避免两个副本同时认为自己是 leader 并对外写入。

问Kafka 支持事务吗？其事务机制是怎样的？
 Kafka 支持事务。它的事务不是传统数据库那种任意读写事务，而是 producer 事务，用于保证“多分区、多 topic 的写入”和“消费位点提交”要么都成功，要么都失败。典型场景是流处理：consumer 消费一条消息后，producer 把处理结果写到下游 topic，并把消费位点一起放进事务提交；下游 consumer 用 `read_committed` 只读已提交事务数据。官方设计文档对 exactly-once 和事务 producer 有完整说明。

问什么是消息的重复消费？
 重复消费就是同一条业务消息被同一个消费逻辑处理了不止一次。典型场景是：consumer 业务处理成功，但 offset 提交失败；consumer 重启后又从旧 offset 再读一遍。Producer 重试也可能让同一条消息被重复写入 topic。

问Kafka 中如何避免重复消费？
 Kafka 本身更偏向 at-least-once，真正避免重复要分层做。Producer 侧开启幂等，避免重试重复写；consumer 侧做业务幂等，比如唯一键、去重表、状态机、幂等写接口；如果是 Kafka Streams 或严格 EOS 场景，再用事务 + `read_committed`。面试时可以明确讲：Kafka 很难单靠 broker 做到业务语义上的“绝对不重复”，最终仍需要消费端幂等。

问Kafka 的 ACK 有哪几种模式？各自的作用是什么？
 `acks=0` 表示 producer 发出就算成功，不等 broker 确认，吞吐最高但最容易丢。`acks=1` 表示 leader 写入本地日志后返回，性能和可靠性折中。`acks=all` 或 `-1` 表示 leader 要等 ISR 副本确认后才返回，是最强的可靠性保证。官方 producer 配置文档明确写了这几种模式及默认值。

问一个 Kafka Topic 被 5 个消费组同时消费（每个 Group 写不同系统），中间某个 Group 偶发失败，但不能重放整条消息（其他组已成功），如何保证该组失败消息的精准重试？同时如何保证幂等性？
 这题的关键是先说明 Kafka 的 offset 是按 consumer group 隔离的。5 个 group 各自维护自己的 offset，所以某一个 group 失败，不会影响其他 group。要做到精准重试，做法通常是：失败 group 不直接回滚整批，而是把失败消息转到它自己的 retry topic 或 dead-letter topic，只重试这一个 group 的失败记录；或者记录失败的 partition+offset 做定向补偿。幂等性靠目标系统唯一键、去重表、状态机，或者把消息主键设计成天然幂等写入。核心思想是“组间隔离，组内精准补偿，目标系统幂等”。

问Kafka 的分区策略有哪些？
 从 producer 角度看，常见分区策略有三类。第一类是显式指定 partition，直接把消息发到某个分区。第二类是指定 key，按 key 经过分区器哈希后路由到固定分区。第三类是不指定 key 和 partition，使用默认分区策略。现代 Kafka 默认逻辑里，无 key 情况下会使用 sticky partitioning 来让一小段时间内的消息尽量落到同一分区，从而提高 batch 效率。官方 producer 文档有这个说明。

问默认分区、粘性分区、自定义分区分别是什么？
 默认分区就是 Kafka 自带的默认 partitioner 逻辑。如果有 key，就按 key 哈希；如果没 key，则倾向使用 sticky 分区，让多条连续消息先聚到一个分区形成更大的 batch，随后再切换分区。粘性分区的核心目标不是“更均匀”，而是“更高的批量发送效率”。自定义分区就是实现自己的 partitioner，比如按 deviceId、业务类型、地域、自定义热点隔离规则等路由到特定 partition。

问Kafka 如何将消息发送到指定的 Partition 中？（底层逻辑、路由规则）
 路由规则可以概括成三层。第一，如果消息显式指定了 partition，就直接发那个 partition。第二，如果没指定 partition 但有 key，就交给 partitioner 对 key 做 hash，通常同一 key 会稳定落到同一分区。第三，如果既没 partition 也没 key，就由默认 partitioner 选择分区，现代实现偏向 sticky 分配来提高批处理效率。Producer 先根据元数据知道 topic 有哪些 partition，再做路由。

问Kafka 消费者的消费策略有哪些？（如 Range、Round-Robin 等）
 消费策略通常指 partition assignor。常见的有 Range、RoundRobin、Sticky、CooperativeSticky。Range 按 topic 分段分配，简单但可能不够均衡；RoundRobin 跨 topic 更平均；Sticky 在尽量均衡的同时尽量少动已有分配；CooperativeSticky 则在 sticky 基础上支持更平滑的协作式 rebalance，减少停顿。官方文档和 Javadoc 都推荐新集群更倾向 cooperative 风格。

问什么是 Kafka 的 Rebalance 机制？
 Rebalance 就是消费组内分区与消费者实例之间的分配关系重新计算和下发的过程。它本身不是异常，而是 group 管理的正常机制。只要组成员变化、分区变化、心跳超时等，都可能触发 rebalance。问题在于 rebalance 期间消费会暂停，所以如果 rebalance 太频繁，吞吐和稳定性就会受影响。

问哪些情况下会触发 Kafka 的 Rebalance？
 常见触发场景有：consumer 实例新增或退出；consumer 宕机；topic 分区数变化；session timeout 导致组协调器判定某实例失联；订阅集合变化；group coordinator 变化。新版本 group coordinator 的实现和可调参数也在持续演进。

问Rebalance 持续时间较长会有什么影响？如何解决？
 影响主要有三点：一是消费停顿，lag 上升；二是重复消费风险增加；三是下游链路会抖动。解决思路是减少 rebalance 频率和缩短单次 rebalance 时间，比如降低 consumer 重启次数、合理设置 session timeout 和 heartbeat、避免消费线程长时间阻塞、采用 CooperativeStickyAssignor、启用静态成员等。

问如何降低 Consumer 重启次数？
 这个题本质上问的是“如何让 consumer 更稳定”。我会从四个方面答。第一，避免业务线程阻塞太久，防止因为心跳超时被踢出组。第二，做好异常隔离，不要因为单条脏数据导致整个进程退出。第三，合理配置 JVM、内存、连接池和下游依赖，减少进程崩溃。第四，用容器探针、优雅停机和静态成员减少不必要的 rebalance 和抖动。

问Kafka 出现消息消费 Lag（消费延迟、消息堆积）时，如何排查问题？怎么处理？
 我会按链路分层排查。先看 producer 侧是不是流量突然暴涨；再看 broker 是否正常，磁盘、网络、ISR、leader 迁移有没有异常；然后看 consumer 侧，是否 rebalance 频繁、业务处理变慢、下游数据库或接口变慢、线程模型不合理、batch 太小、单分区热点。处理手段包括：扩 consumer 实例和分区、加大 batch、优化反序列化和业务逻辑、异步化耗时操作、提高下游吞吐、热点拆分、临时降级非关键消费。核心思路不是一上来就扩机器，而是先定位瓶颈在 producer、broker、consumer、还是下游。

问除了限流和丢弃消息，还有哪些方式可以提高消费者的吞吐量，解决消息积压？
 可以从五个方向做。第一，提升并行度，比如增加 partition 数和 consumer 实例数。第二，提升单实例处理能力，比如批量拉取、批量写库、异步 IO、线程池拆分、减少反序列化和对象创建。第三，优化下游，比如数据库批量写、缓存、连接池、索引优化。第四，降低 rebalance 和 GC 抖动。第五，做热点治理，比如按 key 打散。真正的提升吞吐，通常不是“多消费几条”这么简单，而是端到端优化。

问单分区单消费者实例场景下，如何提高 Kafka 的吞吐量？
 单分区单消费者时，瓶颈不在 Kafka 并行度，而在这个消费者实例本身。提升手段主要是：调大 fetch/batch，减少每条消息固定开销；异步化或并行化业务处理；批量落库；减少序列化和对象分配；优化下游依赖；必要时改设计，把单分区拆成多分区，因为单分区天然把并行上限锁死了。

Kafka、RabbitMQ 如果要监听多个 Topic，需要写多个监听器吗？
 不是必须。无论 Kafka 还是 RabbitMQ，都可以让一个 consumer/监听器订阅多个 topic 或 queue，再在代码里按 topic/tag/type 分发处理。但从工程上看，如果多个 topic 的消费语义、并发模型、重试策略差异很大，通常会拆成多个监听器或多个消费组，便于隔离和运维。

问Kafka 的 Leader 选举策略是什么？
 Kafka 的 leader 选举核心是：由 controller 协调，从合适副本中选出新的 leader，优先保证数据安全和集群一致性，而不是单纯“谁活着谁上”。面试里不要答成 Raft 直接选 leader 那种口径，partition leader 选举更准确的说法是“controller 根据分区元数据、ISR 状态和可选副本条件来决定新 leader”。

问Kafka 的选举机制具体是怎么实现的？（结合 ISR、Controller 等）
 一个 partition 的 leader 不可用后，controller 会感知到 broker/partition 状态变化，然后基于元数据找出该 partition 的可选副本集合，通常优先从 ISR 中挑选新 leader，并更新元数据；随后 producer 和 consumer 刷新元数据，请求转到新 leader。新版 Kafka 中，这些元数据和控制逻辑由 KRaft 控制面统一管理。

问为什么 Kafka 速度快？（从存储、IO、网络等角度分析）
 Kafka 快主要是因为它不是按传统 MQ 那种“每条消息都走复杂确认和随机 IO”的路径来做的。存储上，它以顺序追加日志为核心，顺序写非常适合磁盘和 SSD。IO 上，它充分利用 page cache，读写模式友好。网络上，它支持批量发送、批量拉取和零拷贝。架构上，它通过 partition 把吞吐横向摊开。实现上，producer 和 consumer 都强调 batch。官方设计文档把高吞吐、顺序写和零拷贝都作为核心设计点。

问如果 Kafka 宕机，如何处理？（分 Broker 宕机、Consumer 宕机等场景）
 如果是 broker 宕机，先看是不是 leader 所在 broker。如果不是 leader，通常影响较小；如果是 leader，则会触发 leader 切换，producer/consumer 需要刷新元数据继续工作。前提是副本数和 ISR 设计合理。如果是 consumer 宕机，group 会 rebalance，其他实例接管分区；如果组里只有一个实例，那就只能等它恢复。真正处理时要分清是单 broker 宕机、网络分区、磁盘故障、还是整集群故障，处理手段完全不同。

问Kafka 的延迟消息队列是如何实现的？
 Kafka 本身没有像 RocketMQ 那样天然成熟的延迟消息语义。常见做法有三种。第一，业务层做延迟 topic，比如按分钟或时间桶分 topic，定时消费。第二，消息先写入存储，靠定时任务扫描到期后再投递到真正消费 topic。第三，用 Kafka Streams 或外部调度系统做延迟编排。面试时要明确：Kafka 能实现延迟消息，但通常不是它最强、最原生的能力。

问如何用 Kafka 解耦文件上传与处理流程？
 标准做法是：文件内容不要直接塞 MQ，先落对象存储或文件存储，Kafka 只发文件元数据，比如 fileId、URL、类型、上传人、校验信息、任务参数。下游多个系统分别消费：一个做转码，一个做缩略图，一个做审核，一个做索引。这样文件上传和处理完全解耦，上传接口更快，处理链路也更可扩展。

问Kafka 和 RabbitMQ 有什么区别？
 一句话概括：Kafka 更像分布式日志和事件流平台，RabbitMQ 更像传统业务消息队列。Kafka 强项是高吞吐、可回放、多订阅、流处理生态；RabbitMQ 强项是低延迟、复杂路由、协议丰富、业务场景友好。Kafka 更适合日志、埋点、流式计算、数据平台；RabbitMQ 更适合订单通知、工作队列、复杂业务路由。

问Kafka、RabbitMQ、RocketMQ 三者的区别与适用场景？
 Kafka 偏高吞吐事件流和数据平台，适合日志、埋点、流计算、实时数仓。RabbitMQ 偏业务消息队列，适合低吞吐但复杂路由、延迟敏感、协议多样的场景。RocketMQ 在很多业务中台场景很强，事务消息、延迟消息、顺序消息等业务特性更成熟，适合交易、营销、订单链路。选型不是谁“更强”，而是谁和你的主诉求更匹配。

问为什么选择 Kafka 实现服务通信？
 如果服务通信的核心诉求是高吞吐、广播分发、多下游独立消费、消息保留和回放，那 Kafka 很合适。比如一个采集平台、日志平台、埋点系统、实时风控、数仓链路，用 Kafka 能同时满足异步、解耦、回放和流式处理。它不是替代所有 RPC，而是适合“事件驱动、数据流驱动”的服务通信。

问选型时为什么不选 RabbitMQ 或 RocketMQ？
 不选 RabbitMQ，通常是因为我们更看重吞吐、保留和回放、多系统订阅，而不是复杂路由和业务队列语义。不选 RocketMQ，通常要看团队生态、现有基础设施、社区和运维成熟度。如果我们的核心诉求是日志流、采集流、埋点流、数据平台能力，Kafka 的生态和使用惯性往往更合适。反过来，如果业务强依赖延迟消息、事务消息、顺序业务语义，RocketMQ 可能更舒服。

问为什么使用 Kafka 做异步通信？有考虑过其他 MQ 吗？
 会考虑，而且一定要考虑。选 Kafka 不是因为它“最流行”，而是因为它的设计目标和我们的链路更匹配。我们看重的是高吞吐、可回放、多消费组独立消费、流式处理生态，以及把业务通信和数据平台能力打通。RabbitMQ 和 RocketMQ 都考虑过，但如果场景是高吞吐事件流、埋点、日志、实时计算和多下游订阅，Kafka 更自然；如果是复杂业务路由、小消息事务型队列，RabbitMQ 或 RocketMQ 反而可能更优。

**音视频推流：基于 Boost.Beast 实现轻量级 WebSocket Server 进行推流；方便后端实时观看视频和音频数据**

用 Boost.Beast 起本地 WebSocket Server 做实时预览旁路。视频不走复杂的媒体协议栈，而是把每帧编码成 JPEG 后通过 WS binary frame 推送；音频推 PCM 小块数据。这样后端/前端实现简单、端到端延迟低，适合调试与实时观察。推流链路和 Kafka 主链路解耦，网络慢时采用有界队列和丢帧策略，保证不会反压采集线程。

这块推流链路的目标不是做标准直播协议，而是做一个轻量、低延迟、便于后端实时查看的预览通道。具体实现上，我基于 Boost.Beast 起了一个本地 WebSocket Server。视频侧，眼动场景图在本地先落 MP4，同时把场景帧编码成 JPEG，通过 WebSocket 以二进制帧推给后端；音频侧，采集到 PCM 数据后直接通过 WebSocket 推送。整个推流链路和 Kafka 主链路解耦：Kafka 负责可靠采集和下游消费，WebSocket 负责实时预览；网络慢的时候推流链路允许丢旧帧，不会反压采集主线程。

问整体线程模型和数据流是怎样的？

> 第一层是采集层。视频帧来自眼动 SDK 的 image callback，音频来自 waveIn 回调。采集线程不直接做网络发送，只做轻量封装，把数据交给后面的缓冲队列。第二层是本地缓冲层。视频用了一个很小的环形队列 `TinyRingQ<FrameItem,4>`，采用覆盖策略，满了就覆盖最旧帧。因为这条链路目标是实时预览，不是完整存储，所以我更关心“最新画面”，不关心“每一帧都必须到达”。音频这边是回调直接广播给 WebSocket Hub，同时也落 wav 文件。第三层是编码和推流准备层。视频写盘线程从 `gRing` 里取出场景帧，一方面用 `FFmpegRecorder` 写 MP4，另一方面调用 `encodeToJPEG` 把当前场景帧编码成 JPEG，再通过 `WsHub::broadcastVideo` 广播。音频则是 `waveInProc` 里拿到 PCM buffer 后，直接 `broadcastAudio`，把原始 PCM 广播出去。第四层是 WebSocket 发送层。每个客户端连接对应一个 `WsSession`。session 内部维护自己的发送队列 `q_`，并且通过 `writing_` 保证同一个 session 的 `async_write` 串行执行。这样多客户端互不影响，慢客户端只会堆自己的队列，不会阻塞其他客户端和采集线程。

问为什么选 WebSocket，不用 RTSP/RTMP/WebRTC？

> 目标是“采集端给后端一个轻量级实时预览能力”，不是做标准直播平台，所以优先级是实现简单、接入方便、开发成本低。WebSocket 基于 HTTP 升级，后端和前端都容易接，Boost.Beast 也比较轻量。相比之下，RTSP/RTMP/WebRTC 都更偏标准媒体协议栈，功能更强，但复杂度也明显更高。尤其 WebRTC 还要处理 ICE、STUN、TURN、SDP 这些问题，对我们这个本地内网预览场景来说过重了。预览链路用 WebSocket，保证快速落地和低耦合；如果后续场景升级到公网、多客户端、大带宽压力，再考虑切到 H264/Opus 或 WebRTC。

问传的是什么数据，讲清协议

> 视频侧：我把场景图编码成 JPEG，然后通过 WebSocket 二进制帧发送。消息头是一个简单的自定义头部，总共 9 字节，第 1 字节表示通道类型，比如 `CH_SCENE=2`，后 8 字节是时间戳 `ts`。后面 payload 就是 JPEG 二进制数据。
>
> 音频侧：PCM 也是一样的封装方式，第 1 字节是 `CH_PCM=3`，后 8 字节是时间戳，然后后面拼接 PCM 原始采样数据。
>
> 所以整个推流协议其实很轻量，本质上就是：
>  `[type(1B) + ts(8B) + payload(NB)]`
>
> 这样做的好处是简单直观，后端拿到数据后按 type 区分视频和音频，按 ts 做简单同步就可以了。

问为什么视频推 JPEG，音频推 PCM，不直接推 H264/AAC

> JPEG 的优点是解码简单，后端或者前端很容易直接显示；PCM 的优点是零编码延迟、处理链路简单，适合快速接入和分析。这样整个链路的实现复杂度很低。JPEG 压缩效率不如 H264，PCM 体积也比 AAC/Opus 大很多，所以这套方案不适合公网、大规模并发或者带宽很敏感的场景。它更适合内网、本地采集端到后端的实时预览。如果后面带宽和并发成为瓶颈，我的演进方向会是： 视频从 JPEG 升级到 H264/H265，音频从 PCM 升级到 AAC/Opus，必要时进一步演进到 WebRTC。

问“Beast 这块是怎么保证线程安全的

> Beast/WebSocket 一个很关键的约束是：**同一个 session 的写操作必须串行化，不能并发 async_write**。这点我代码里是通过 `q_ + writing_` 实现的。
>
> 具体逻辑是这样的：
>
> 当外部线程调用 `pushScene` 或 `pushPcm` 时，不会直接写 socket，而是先把待发送数据封装成 `shared_ptr<vector<uint8_t>>`，然后通过 `net::post(ws_.get_executor(), ...)` 投递到这个 session 的 executor 上。投递进去之后，先加锁把消息放进 `q_`，再调用 `flush()`。
>
> `flush()` 会先判断两个条件：
>  如果当前已经有写操作在飞，也就是 `writing_ == true`，那就直接返回； 如果队列为空，也直接返回；只有在“当前没有写、而且队列非空”时，才会取队头消息调用 `async_write`。等 `async_write` 完成后，在 `on_write()` 里把队头 pop 掉，`writing_ = false`，然后再继续 `flush()` 下一条。串行写队列，这样就满足了 Beast 对写串行化的要求。

问“多客户端会不会互相影响”

> 不会，因为我不是一个全局发送队列给所有客户端公用，而是每个连接一个 `WsSession`，每个 session 自己有自己的发送队列 `q_` 和自己的 `writing_` 状态。Hub 只是做 session 管理和广播分发。比如 `broadcastVideo` 时，实际上是遍历所有 session，逐个调用 `pushScene`。后续每个 session 的发送节奏由它自己的写队列决定。第一，慢客户端不会拖累快客户端；第二，网络发送和采集线程解耦，采集线程只负责广播调用，不会阻塞在某个客户端的 socket 上。

问怎么处理背压，网络慢怎么办”

> **推流是预览旁路，优先保实时，不保完整。**
>
> 视频侧最关键的背压点在 `TinyRingQ<FrameItem,4>`。这个队列容量只有 4，而且满了采用覆盖策略，也就是写新帧时会覆盖最旧帧。这意味着如果处理或发送速度跟不上，我会主动丢掉旧画面，始终尽量保留最新帧，保证预览看到的是“当前状态”，而不是很久以前积压的旧画面。
>
> 

问“视频为什么还要本地写盘，同时再推 JPEG”，你可以这么答

> 这是两条不同目标的链路。
>
> 本地 MP4 写盘是为了完整保存采集过程，后面可以做回放、分析、证据留存；JPEG 推流是为了实时预览，方便后端立即看到当前场景。同一个 `FrameItem`，既走 FFmpeg 编码写 MP4，也走 JPEG 编码推给 WebSocket。
>  这样可以最大化复用采集到的原始图像，避免采集层重复取数据。

问你的时间戳是怎么用的，音视频怎么对齐”，你可以这样答

> 我在视频和音频包头里都带了一个 8 字节时间戳 `ts`。视频里是场景帧的时间戳，音频里是当前 PCM chunk 的采集时间。后端拿到这两个时间戳后，可以做简单的软同步。链路目标是实时预览，不是严格的广播级 lip-sync，所以不需要非常复杂的同步策略。通常后端按 `ts` 做近似对齐就够用了。如果以后需要更强同步，可以进一步引入统一时钟源和更严格的播放缓冲策略。

问“你现在这套推流链路的主要优点是什么”，你可以总结成这四点

> 第一，轻量，基于 Beast + WebSocket，开发和接入成本低。 第二，低耦合，和 Kafka 主链路分开，互不影响。第三，低延迟，视频采用 JPEG 即时推，音频采用 PCM 原样推，编码开销低。第四，可降级，视频队列有界且覆盖旧帧，网络慢时不会把采集主线程拖死。

问“主要缺点和后续演进方向是什么”

> 第一，带宽利用率不高。JPEG 和 PCM 相比 H264/AAC 更占带宽。音频链路的背压还可以继续加强。视频这边已经有 TinyRingQ 覆盖策略，但音频 session 队列还可以加上限和丢包策略。协议比较轻量，适合内网预览，不是标准流媒体协议。如果要扩展到公网、多客户端或更复杂场景，需要演进到更标准的媒体方案。
>
> 后续演进方向我会分三步：先把 session 队列做有界化和丢弃策略统一；再把视频改为 H264、音频改为 AAC/Opus；如果再往上走，就切到 WebRTC。
